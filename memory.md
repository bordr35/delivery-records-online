# Project Memory

- Delivery Records is intended to become a polished, reliable field command app for gig-delivery work, not a throwaway static form.
- The product goal is practical shift readiness: quick capture during work, strong saved records, clean human-readable exports, and a violation/dispute packet that can actually be used with support.
- JSON backup is only for restoring browser/device data. It should never be presented as the main way to fight a violation.
- The violation workflow should prioritize plain-English appeal text, evidence timelines, proof checklists, order/case IDs, money-at-stake, and downloadable/copyable TXT/PDF-style outputs.
- Keep the app static-hosting friendly for now, but make it feel like a serious product: dependable navigation, autosave, search/filter, useful exports, mobile-first design, and explicit cache/update behavior.
- Next product direction: make Violation Packet a guided case builder and support normal proof screenshots/photos with private in-browser storage, thumbnail previews, notes, and packet evidence inventory.
- v2.1.0 direction: proof shots should support normal mobile screenshots/photos, local thumbnail review, "what this proves" notes, downloadable image copies, and inclusion in the Violation Packet evidence inventory.
- Current product vision: cloud sync is core, not optional forever. Use encrypted cloud sync so phone/desktop records and proof shots become one dataset, then use that dataset for delivery strategy analysis across apps, zones, miles, pay, issues, disputes, and proof habits.
